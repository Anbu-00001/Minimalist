# Research: "certificate-carrying" verification beyond math/logic — code, summarisation, factual QA

Research date: 2026-07-09. Scope per request: extend AMDA's "answer ships with a
machine-checkable certificate, not just a confidence score" pattern (currently math via
program-aided re-derivation, `agent/verifiers.py::extract_expression`/`run_expression`;
logic via CSP re-derivation, `parse_logic_translation`/`solve_logic_csp`) to
code_generation/code_debugging, text_summarisation, and factual_knowledge. Unlike the
collect-only files in this directory, this one is explicitly asked to end in a ranked
recommendation, so analysis and verdicts are threaded through rather than held to the end.
Verbatim quotes are marked with quotation marks and a source; paraphrases are marked
"(paraphrase)"; anything not independently confirmed is marked **UNVERIFIED**.

**Grounding fact that shapes every verdict below**: AMDA's agent process currently has
**zero ML-framework Python dependencies**. `agent/local_llm.py` talks to a separately
compiled `llama-server` binary over HTTP (`httpx`/`openai` client only); `requirements.txt`
is `openai`, `httpx`, `python-constraint2`. The `Dockerfile` is `python:3.12-slim` +
`libgomp1` + those three pip packages + the llama.cpp binary + baked GGUF weights. Any
candidate that needs `torch`/`transformers` is not "add a small model," it's "add a new
dependency class to a container that has deliberately avoided one," inside a 4 GB RAM / 2
vCPU grading box that already spends ~2.5 GB on the primary model (Qwen3-4B Q4_K_M, `-c
2048`, per `research/VERDICTS.md` V20, sourced from the organizer's Participant Guide).
That headroom arithmetic (roughly ≤1 GB genuinely free, shared with Python/OS overhead)
is the binding constraint for sections 2 and 3 below, more than raw model parameter count.

## 1. Property-based test synthesis for LLM-generated code

### 1.1 What EvalPlus actually does (verified from the paper PDF, `arxiv.org/pdf/2305.01210`, NeurIPS 2023)

Full title: *"Is Your Code Generated by ChatGPT Really Correct? Rigorous Evaluation of
Large Language Models for Code Generation"* (Liu, Xia, Wang, Zhang — UIUC/Nanjing).
Verbatim from §2.1, "Automated Test Input Generation":

> "EvalPlus first uses ChatGPT to generate a set of high-quality seed inputs for later
> mutation. Following Figure 2, we construct a prompt using (i) the ground-truth solution
> of the problem for ChatGPT to inspect; (ii) a set of test inputs as demonstration; and
> (iii) an instruction to encourage ChatGPT to come up with interesting inputs... By using
> ChatGPT as an automated generation engine, we can generate inputs that are valid even
> under semantic constraints... However, ChatGPT is unsuitable for large amounts of
> automated test generation due to undesired speed and cost of querying such a large
> model. To address this, we perform type-aware input mutation starting from high-quality
> seed inputs generated by ChatGPT."

So the pipeline is two stages, and **only the first stage needs an LLM**:
1. **LLM-seeded corner cases** — ChatGPT, shown the ground-truth solution (!) plus a few
   example inputs, proposes "interesting"/"difficult"/"corner-case" inputs. EvalPlus's own
   paper says this needs ChatGPT-class capability partly *because* the seed generator is
   given the ground-truth implementation to inspect — a "read the reference solution and
   attack it" strategy AMDA does not have available (there is no gold answer to hand the
   model).
2. **Type-aware mutation** — deterministic, pure-Python, zero LLM calls. Table 1 (verbatim,
   transcribed from the PDF) gives the exact per-type mutation rules: `int|float` → `x±1`;
   `bool` → random boolean; `str` → remove/repeat/replace a substring; `List` →
   remove/repeat/insert-or-replace an element with a recursively mutated one; `Tuple`/`Set`
   → mutate as a List then re-wrap; `Dict` → remove/update/insert a key-value pair. This
   step is exactly Hypothesis's territory (§1.4 below) and needs no model at all.

Scale: "for each task, based on around 30 ChatGPT-generated seed inputs... we run
type-aware mutation to generate 1000 additional inputs using a one-hour budget" (§3,
Evaluation). That "one-hour budget" line is itself informative: even EvalPlus, with a
frontier model doing the seeding, treats large-scale mutation as an offline,
minutes-to-hours process for building a *benchmark*, not a per-inference-call check.
**Directly relevant finding, stated by the authors**: "by inspecting the ground-truths in
the original HumanEval, we found over 10% of them are incorrectly implemented" — even
hand-authored gold answers have a nontrivial wrong-test-suite rate, a useful calibration
point for how much to trust any single generated verification signal, including AMDA's own.

### 1.2 What CodeT actually does (verified from the paper PDF, `arxiv.org/pdf/2207.10397`, Microsoft, EMNLP 2022)

*"CodeT: Code Generation with Generated Tests"* (Chen, Zhang, Nguyen, Zan, Lin, Lou, Chen).
Mechanism, verbatim from §2.1/§2.2:

> "We randomly select a pair (x, y) from the set of all possible pairs D = {(x, y)|x ∈
> X, y ∈ Y}. We then try to execute the code solution x on the test case y... If (x, y) is
> a hypothetical inlier, we collect all other pairs from D that agree with (x, y), forming
> a set S called consensus set... We score the consensus set as f(S) = |Sx||Sy|... The
> intuition is that the more pairs that agree with the hypothetical functionality, the
> more likely this functionality is correct."

Test cases themselves are generated by the **same** pretrained LM, prompted with the
docstring/context plus an instruction template ending in a bare `assert` line so the model
completes it as an input→expected-output pair (Figure 2 in the paper shows the exact
prompt shape: context + `pass` placeholder + `# check the correctness of {entry point}` +
`assert`). Reported gains are large — code-davinci-002 on HumanEval: 47.0% → 65.8% pass@1
(+18.8 points); on CodeContests, 0.7% → 2.1% pass@1 — but the mechanism is fundamentally
**N-candidates × M-tests, both sampled at temperature 0.8**, with N up to 100–1000 in their
own experiments (Table 2/3). This is CodeT's design point, not an incidental detail: the
whole "dual execution agreement" signal comes from having *multiple different code
candidates* to cross-check against each other's test-passing pattern. **AMDA has exactly
one candidate answer to verify at a time** (a single local-model completion that already
passed the router), so CodeT's core mechanism — consensus across many samples — does not
transfer without either (a) generating several more full completions (real latency cost,
see §1.6) or (b) degrading to "generate tests, run the one candidate against them," which
is no longer CodeT, just self-generated assertion testing (closer to the CRITIC pattern
already covered in `research/program_aided_verification_patterns.md` §2).

### 1.3 AlphaCode's test generation (WebSearch-derived, not independently PDF-verified this pass — treat as medium confidence, consistent across multiple secondary sources)

AlphaCode (DeepMind, 2022) mutates existing *hidden* test inputs (bit-flips on binary
inputs, integer increment/decrement, character swaps in strings) and **filters mutants by
running 30 known-correct solutions and keeping only inputs where all 30 agree** on the
output — i.e., it also needs multiple independent correct implementations to validate its
own generated tests, same structural dependency as CodeT. Source:
https://victordibia.com/blog/alpha-code/ (secondary summary) cross-referenced against the
paper PDF at https://storage.googleapis.com/deepmind-media/AlphaCode/competition_level_code_generation_with_alphacode.pdf
(not independently re-fetched and quoted verbatim in this pass — **UNVERIFIED** at the
verbatim-quote level, though the general shape is consistent with the CodeT/EvalPlus
literature and with AlphaCode's public reputation). Not usable for AMDA for the same
multi-implementation reason as CodeT.

### 1.4 Hypothesis Ghostwriter — the one genuinely zero-model, zero-gold-test option (verified: GitHub/PyPI + WebSearch cross-check)

- Library: `hypothesis` (PyPI: https://pypi.org/project/hypothesis/, repo:
  https://github.com/HypothesisWorks/hypothesis). Pure Python, no ML dependency, MIT-style
  license (standard OSS testing library; already a very common transitive dependency in the
  Python ecosystem, unlikely to conflict with anything).
- The **Ghostwriter** feature (`hypothesis write`) inspects "the function name, argument
  names and types, and docstrings" and "produces a single semantically-meaningful test per
  function" (paraphrase, consistent across the docs and multiple independent summaries
  found via WebSearch — direct WebFetch of the ghostwriter docs page itself 404'd/returned
  empty content in this pass, so treat the exact wording as **UNVERIFIED** paraphrase, but
  the feature's existence and CLI surface (`hypothesis write --roundtrip ...`, `hypothesis
  write --idempotent sorted`, `ghostwriter.fuzz(...)`) is corroborated by the PyPI page,
  the GitHub repo, and independent third-party write-ups).
- Ghostwriter modes that need **no human-authored property at all**:
  - `fuzz()` — generates inputs from the function's type hints/signature and just checks
    the function **doesn't crash** on a wide, adversarially-search-generated input space
    (edge values, empty collections, negative numbers, unicode edge cases, etc. — this is
    exactly the standard `hypothesis.strategies` search space, the same machinery
    EvalPlus's Table 1 hand-rolls type-aware mutators for).
  - `idempotent()` — checks `f(x) == f(f(x))`, useful for the sort/dedup/normalize shape of
    function some code_generation prompts ask for.
  - `roundtrip()` — checks `decode(encode(x)) == x`, useful when the task pairs two
    functions (e.g. serialize/parse).
- Cost: pure CPU, no network, no extra model call. Hypothesis's default example budget
  (100 examples/test) runs in well under a second for arithmetic/string/list functions —
  comfortably inside the 30s/task and 25s `REQUEST_TIMEOUT_S` budgets even stacked on top
  of the local model's own generation and the existing `_run_python` check.

### 1.5 Metamorphic Prompt Testing — the closest hit to "verify a single existing answer, no gold tests, no retrieval" (verified: paper PDF/abstract at `arxiv.org/abs/2406.06864`)

*"Validating LLM-Generated Programs with Metamorphic Prompt Testing"*. Verbatim abstract:

> "Our intuitive observation is that intrinsic consistency always exists among correct
> code pieces but may not exist among flawed code pieces, so we can detect flaws in the
> code by detecting inconsistencies. Therefore, we can vary a given prompt to multiple
> prompts with paraphrasing, and to ask the LLM to acquire multiple versions of generated
> code, so that we can validate whether the semantic relations still hold in the acquired
> code through cross-validation. Our evaluation on HumanEval shows that metamorphic prompt
> testing is able to detect 75 percent of the erroneous programs generated by GPT-4, with
> a false positive rate of 8.6 percent."

Mechanism (from the paper body, WebFetch-derived — flag as paraphrase-with-quotes, not an
independently re-verified PDF read in this pass): "Taking prompts with the same meaning,
the LLM should generate programs with the same semantics" — they paraphrase the original
prompt into **5 total variants (1 original + 4 paraphrases)**, generate code for each, then
use fuzzing (Python-AFL, ~20 random inputs) to check that all 5 programs agree on outputs;
majority disagreement flags the code as erroneous. **This is a genuinely no-gold-test,
no-retrieval, single-model-capable technique** — GPT-4 was used for both paraphrasing and
code generation in the paper, so nothing stops a single local model from playing both
roles. The catch for AMDA is cost: **5 full code-generation calls per task**, not 1. At
whatever per-call latency the local model runs at (VERDICTS.md doesn't give an exact
figure, but the 25s `REQUEST_TIMEOUT_S` and 30s/task organizer rule imply local generations
are budgeted in the low-single-digit-seconds to ~10s range for these task sizes), 5×
generation is very likely to blow the per-task time budget on its own, before the fuzzing
step. A **cut-down 2-version variant** (original candidate + one re-generation from a
paraphrased instruction, compare outputs on a handful of Hypothesis-generated inputs) is
structurally the same idea at roughly 2× generation cost instead of 5× — untested in the
literature (the paper's own ablation floor is 3 versions, not 2: "we consider odd numbers
only here so that our cross validation always works with a clear majority" — paraphrase),
so a 2-version cut would trade away the majority-vote decisiveness the paper relies on and
its detection rate would not be expected to hold at 75%.

### 1.6 Differential testing without an oracle (WebSearch-derived context, DiffSpec paper `arxiv.org/abs/2410.04249` not independently PDF-verified this pass — **UNVERIFIED** at the verbatim level)

The general framing (paraphrase, consistent across secondary sources): "differential
testing... eliminates the need for explicit oracles... running the same input on multiple
implementations and flagging behavioral mismatches." This is the same
multiple-candidates-required structure as CodeT/AlphaCode (§1.2–1.3) — not usable with a
single candidate answer without generating more candidates, which is again a latency
trade-off, not a free win.

### 1.7 What this means for AMDA's current code verifier specifically

Read together with `agent/verifiers.py`, there is a concrete, low-effort, high-confidence
finding here, independent of any of the papers above: **the current check barely exercises
the candidate function's logic at all.** `extract_python_code` returns the longest
contiguous block that parses as Python; `_run_python` then executes that block **as a
standalone script**. For the actual task shape in this project (confirmed by
`test_io/practice_tasks.json`, e.g. practice-08: *"Write a Python function that returns the
second-largest number in a list, handling duplicates correctly."* — a bare NL spec, no
docstring examples, no provided test cases) the extracted block is typically just a `def
f(...): ...` with **no call site**. Running `def f(...): ...` as a script raises nothing
regardless of whether the function body is correct — Python only complains about syntax
errors and whatever executes at module scope (imports, any stray top-level code). So the
current "pass" verdict for code tasks is much closer to "the answer contains syntactically
valid Python that defines something" than "the answer's logic runs successfully," and
Hypothesis's `fuzz()` mode (§1.4) — which actually **calls** the extracted function under a
search-generated battery of inputs — is a direct, mechanical fix for exactly this gap, at
effectively zero marginal latency and zero new dependency risk (pure Python, tiny package).

## 2. NLI/entailment-based summary faithfulness checking

### 2.1 Vectara HHEM (verified: HF model card fetch)

- Current open model: **HHEM-2.1-open**, ~0.1B (100M) parameters, Apache 2.0 license
  (commercial use and hackathon submission both fine per the license terms as fetched).
  GitHub leaderboard: https://github.com/vectara/hallucination-leaderboard.
- Input format is a plain NLI pair, not a full document+summary pipeline requiring extra
  scaffolding: `List[Tuple[str, str]]` of (premise, hypothesis), with the model's own
  training template "Determine if the hypothesis is true given the premise?" — i.e. usable
  directly as (source_document, generated_summary) → entailment score.
- Reported cost/latency (from the model card, fetched verbatim): **"around 1.5 second for a
  2k-token input on a modern x86 CPU"** using fp32, **"less than 600MB RAM."** That RAM
  figure is a real problem for AMDA's headroom (§ intro) — 600MB is roughly half-to-all of
  the free RAM left after the primary model, before accounting for a second Python process
  needing `transformers`+`torch` at all (see intro's dependency-footprint point; HHEM is
  distributed as a HF `AutoModelForSequenceClassification`, so it pulls in the same heavy
  stack). A quantized/int8/ONNX build would shrink this, but no such build was confirmed in
  this pass — flag as **UNVERIFIED** whether one exists officially.
- Reported accuracy (model card, verbatim numbers): RAGTruth-QA 74.28% balanced accuracy,
  **RAGTruth-Summ 64.42%**, AggreFact-SOTA 76.55%. The RAGTruth-Summ number — the one
  closest to AMDA's actual task (checking a summary against its source) — is the weakest of
  the three, worth noting: this is not a slam-dunk accuracy story even before the RAM
  question.

### 2.2 TinyLettuce / LettuceDetect (verified: HF blog fetch + GitHub docs fetch)

- Project: LettuceDetect (KRLabsOrg), TinyLettuce variants are distilled onto "Ettin"
  encoders (LightOn) at **17M, 32M, and 68M parameters**. MIT license. `pip install
  lettucedetect`.
- Verbatim/paraphrase-with-numbers from the HF blog
  (https://huggingface.co/blog/adaamko/tinylettuce): "TinyLettuce-17M... reaches 90.87% F1
  on synthetic test data, outperforming GPT-5-mini (83.69%)..." — but on the **real,
  non-synthetic RAGTruth benchmark, TinyLettuce-68M reaches 74.97% F1**, a substantially
  smaller margin over the synthetic-data headline number. This is the same
  synthetic-vs-real gap pattern worth flagging honestly rather than quoting the flashier
  90.87% figure as if it were the real-world number.
- **Format mismatch risk, confirmed from the docs, not resolved**: the API and every
  documented usage example is `context + question + answer` (RAG-QA shaped —
  `detector.predict(context=[...], question="...", answer="...")`), not a plain
  document+summary pair. Nothing in the fetched docs (blog post or
  `docs/TINYLETTUCE.md`) shows a summarization-specific usage example or reports a
  summarization-specific benchmark number. Using it for AMDA's text_summarisation category
  would mean feeding a synthetic/empty "question" field (e.g. "Summarize the following
  faithfully") — **this is plausible but untested**; the published F1 numbers are all for
  the QA shape the model was actually trained/evaluated on, not for bare summarization.
- On-disk size and CPU latency: not confirmed numerically in this pass (model card fetch
  for the specific 17M checkpoint 401'd). Given 17-68M-parameter transformer encoders, a
  rough order-of-magnitude estimate (not a verified figure) is tens-to-~150MB on disk in
  fp32 — much smaller than HHEM's 600MB RAM figure, and the better RAM fit of the two
  options — but still requires the same `transformers`/torch-class dependency addition
  the intro flags as a real integration cost for this specific codebase, unless a smaller
  ONNX Runtime path is used instead (see §5).

### 2.3 SummaC (verified: paper abstract via WebSearch + GitHub repo fetch)

*"SummaC: Re-Visiting NLI-based Models for Inconsistency Detection in Summarization"*
(Laban, Schnabel et al., TACL 2022, arXiv:2111.09525). Two variants: SummaC-ZS (zero-shot,
sentence-level NLI score aggregation via max/mean, no extra training) and SummaC-Conv
(learns a light aggregation layer). Reported "state-of-the-art results with a balanced
accuracy of 74.4%" on their own 6-dataset SummaC benchmark at the time of publication
(2021/2022 — likely surpassed since by HHEM/newer work, not re-checked). GitHub
(https://github.com/tingofurro/summac) confirms `pip install summac`, `device="cpu"`
support, and that scoring is literally:
```python
from summac.model_summac import SummaCConv
model = SummaCConv(models=["vitc"], device="cpu")
score = model.score([document], [summary])
```
i.e. the library already implements the exact "split summary into sentences, run each
against the source with an off-the-shelf NLI model, aggregate" pipeline by hand — the
underlying NLI model (`"vitc"`) is not confirmed by parameter count in this pass
(**UNVERIFIED** size), but SummaC's method is a documented, citable template for rolling a
custom, smaller version with any NLI checkpoint of choice (see §2.4), rather than needing
this specific package's bundled model.

### 2.4 FactCC / QAFactEval (verified: WebSearch summaries; not independently PDF-fetched this pass)

- FactCC (Salesforce, 2019/2020, https://github.com/salesforce/factCC): BERT-base-sized
  (~110M params, i.e. roughly the same weight class as HHEM), weakly-supervised via
  rule-based sentence transformations, reports **72.65% accuracy on CNN/DailyMail**
  (paraphrase from search summary). Older architecture, no confirmed modern
  quantized/small build found in this pass.
- QAFactEval (2021): a **pipeline**, not a single model — question generation + question
  answering + answer-overlap scoring, "14% average improvement over previous QA-based
  metrics" (paraphrase). This needs at least two extra model calls (a QG model and a QA
  model) per summary, layered on top of AMDA's primary local model. **Not feasible** inside
  the stated 30s/task, 2-vCPU budget — flagged explicitly rather than left implicit, per
  the instruction to be honest about infeasibility.

### 2.5 Generic tiny NLI cross-encoders (WebSearch-derived; a DIY-SummaC alternative)

Off-the-shelf sentence-pair NLI cross-encoders exist well below HHEM's size, e.g.
`all-MiniLM-L6-v2`-class models (~22M params, reported "roughly five times smaller than
BERT-base," ~14,000 sentence encodings/sec on a standard CPU for the embedding variant —
paraphrase from search summary, not independently re-verified for the *cross-encoder NLI*
variant's exact throughput). These are a lower-risk building block than either HHEM or
TinyLettuce specifically **because** they use the plain premise/hypothesis interface
SummaC's method already assumes, avoiding both HHEM's RAM figure and TinyLettuce's
QA-shaped API uncertainty — at the cost of writing the sentence-splitting + max/mean
aggregation loop yourself (well-documented in the SummaC paper as the reference algorithm,
~30-50 lines of Python) rather than getting it for free from a library call.

## 3. Claim-decomposition + verification for factual QA — largely NOT feasible without retrieval, confirmed honestly

### 3.1 FActScore itself requires retrieval (verified: abstract + secondary summaries)

*"FActScore: Fine-grained Atomic Evaluation of Factual Precision in Long Form Text
Generation"* (Min et al., EMNLP 2023, arXiv:2305.14251). The metric is explicitly defined
as: decompose into atomic facts, then "assigning a binary label to each atomic fact" by
checking it **"against a reliable knowledge source"** — their automated estimator "uses
retrieval and a strong language model" (paraphrase, consistent across all sources found).
**There is no retrieval-free version of FActScore itself.** AMDA has no permitted knowledge
base or search API at grading time (only the local model and the Fireworks proxy for the 5
allowed models), so the FActScore mechanism as published is not implementable as stated —
this negative is stated plainly per the instruction to be honest about infeasibility.

### 3.2 SelfCheckGPT — the "self-consistency as substitute for retrieval" technique the task asked about, confirmed real (verified: abstract fetch, `arxiv.org/abs/2303.08896`)

Verbatim abstract fragment: "SelfCheckGPT... is a simple sampling-based approach that can
be used to fact-check the responses of black-box models in a zero-resource fashion... if an
LLM has knowledge of a given concept, sampled responses are likely to be similar and
contain consistent facts. However, for hallucinated facts, stochastically sampled responses
are likely to diverge." Five consistency-measurement variants reported (BERTScore, QA,
n-gram, NLI, prompting) — headline numbers (AUC-PR etc.) were not independently re-verified
in this pass (the specific figures were not retrievable from the abstract-only fetch
available; flag as **UNVERIFIED** at the exact-number level, though the mechanism and
zero-resource framing are solid). This is a real, citable answer to "is self-consistency a
documented substitute for retrieval" — yes, it is a published, reasonably well-known
technique, not a made-up idea.

### 3.3 Atomic Self-Consistency (ASC) — FActScore-style decomposition + self-consistency, no retrieval, but not designed as a single-answer verifier (verified: paper PDF, `aclanthology.org/2024.emnlp-main.706.pdf`, EMNLP 2024)

Confirmed via direct PDF read. ASC's own framing of the cost problem, verbatim (§2.3,
discussing a concurrent approach that tried full pairwise LLM-based atomic-fact
consistency checking): that method is **"extremely expensive requiring O(n × m) LLM
calls"** where n and m are both counts of samples/facts. ASC's own method reduces this by
using cheap sentence tokenization instead of LLM-based fact extraction ("we used individual
sentences within a candidate generation as its atomic facts... using sentence tokenization
models," citing Bird et al. 2009 — i.e. plain NLTK-style sentence splitting, not a model
call) and clustering, but it is fundamentally a **generation-quality** technique (merge the
best sub-parts of m independently-sampled responses into one better final answer), not a
verifier that scores one already-produced answer as pass/fail. It still requires **m
independent stochastic samples from the model per question** (m is not pinned down exactly
in the pages read, but the framing throughout is "multiple stochastic samples," consistent
with SelfCheckGPT-scale sampling counts, i.e. more than the 1-2 extra generations that would
fit cheaply in AMDA's budget).

### 3.4 ACPO — explicitly a training-time method, not applicable (verified: abstract fetch, `arxiv.org/abs/2505.09039`)

*"Atomic Consistency Preference Optimization"* uses the same atomic-consistency-across-samples
signal, but verbatim: it is "a self-supervised **preference-tuning** method" used to build
training data pairs for model alignment. Confirmed **not** an inference-time verifier — it
fine-tunes a model rather than checking a single answer. Ruled out for AMDA, which does no
training.

### 3.5 Net honest answer to research question 3

Self-consistency across multiple local generations, as the task description itself
suggested as a "weak substitute," **is a real, published idea** (SelfCheckGPT, ASC) rather
than a fabrication — but every documented instance of it costs multiple (the literature
clusters around 5-20, not 2-3) full extra generations per question to get a meaningful
signal, and AMDA already has direct, in-repo evidence that a *cheap*, single-signal
confidence proxy for open-ended categories doesn't reliably calibrate: `research/VERDICTS.md`
V17 records that the already-implemented free logprob-confidence gate was **disabled**
after a 26-task calibration showed "pass/fail mean-logprob distributions overlap." A
self-consistency-based gate would face the same fundamental calibration question (where's
the threshold?) with a much higher latency cost per sample than logprobs, which are free.
No FActScore-without-retrieval substitute was found that both (a) avoids external
knowledge and (b) fits the per-task latency budget. This looks like a genuine "not
feasible under these specific constraints" conclusion, not a gap in the search.

## 4. General 2025-2026 "verifiable answer bundle" / self-certifying output frameworks

Searched specifically for recent academic framing of LLM outputs shipping with
proof-of-work/verification artifacts rather than confidence scores, per the request.

- **"Tool Receipts, Not Zero-Knowledge Proofs: Practical Hallucination Detection for AI
  Agents"** (arXiv:2603.10060, fetched, WebFetch-derived summary — treat as
  paraphrase-with-partial-quotes, not independently re-verified verbatim from the raw PDF
  in this pass). Proposes lightweight "receipts" (deterministic hashing / HMAC-style
  records of what a tool call actually returned) as a practical alternative to heavy
  cryptographic proof for catching agents that fabricate tool outputs. Conceptually this is
  the closest thing in the recent literature to naming AMDA's own existing math/logic
  verifiers as a pattern: independently re-derive/re-execute and keep the receipt, rather
  than trust the model's self-report. Not a new technique to adopt — AMDA already does
  this for math and logic — but useful as external validation that the general shape is a
  recognized, actively-published idea in 2026, not an idiosyncratic design choice.
- **"Anytime-Valid Answer Sufficiency Certificates for LLM Generation via Sequential
  Information Lift"** (Sequential-EDFL, arXiv:2510.06478, abstract + methodology fetched).
  A real, formally-grounded statistical certificate computed purely from the model's own
  log-likelihoods (comparing the full model against deliberately weakened "skeleton"
  baselines), cheap to compute (no external verification, no ground truth). **Explicitly
  not a correctness certificate** — verbatim: the guarantee "does not imply delta control
  of factual incorrectness or hallucinations," and even their own correctness-gated variant
  leaves "10.9 percent of stopped sequences remain incorrect." Interesting, honestly
  irrelevant to AMDA's actual need (an accuracy-gate pass), included because it was the
  closest hit to the literal search terms and the paper itself flags the limitation clearly.
- **"Proof-Carrying Numbers (PCN): A Protocol for Trustworthy Numeric Answers from LLMs via
  Claim Verification"** (arXiv:2509.06902, abstract fetched verbatim). Verbatim: "numeric
  spans are emitted as claim-bound tokens tied to structured claims, and a verifier checks
  each token under a declared policy (e.g., exact equality, rounding, aliases, or tolerance
  with qualifiers). Crucially, PCN places verification in the renderer, not the model...
  This separation prevents spoofing and guarantees fail-closed behavior." Model-agnostic in
  principle (verification lives outside the model), but the abstract does not specify what
  the verifier actually checks numeric claims *against* — for AMDA's math category this
  slot is already filled by the existing program-aided re-derivation; for other categories
  there is no obvious "ground truth to check against" in this protocol either, so it's a
  useful naming/framing reference more than a new implementable mechanism.
- **"Proof-Carrying Certificates for LLM Pipelines: A Trust-Boundary Architecture"**
  (arXiv:2605.16407, abstract fetched verbatim). Requires a full **Lean 4** kernel
  type-check plus "sorry-free transitive axiom audit," a compiled Lean 4 reference artifact
  covering 22 certificate types. Explicitly ruled out: this targets regulated/high-stakes
  formal-verification deployments, not a resource-constrained hackathon container, and
  learning + integrating Lean 4 before 2026-07-11 is not realistic.
- **zkLLM / zero-knowledge proof of LLM inference** (mentioned in search results, not
  independently fetched — **UNVERIFIED** at the source level, treat as background
  awareness only): reported (paraphrase from search snippet) to run "thousands of times
  slower than plain inference." Ruled out on cost grounds alone even if verified.
- **LLMCert family** (LLMCert-B/C/T, ICLR/AISTATS/ICLR-workshop 2025-2026, WebSearch
  summary only, **UNVERIFIED** beyond snippet level): statistical certification of a
  *model's* behavior over a *distribution* of prompts (bias, knowledge comprehension,
  tool-selection robustness) — a different unit of certification than "is this one answer
  to this one task correct," not directly applicable to AMDA's per-task verification need.

**Net answer to research question 4**: nothing found names or generalizes AMDA's exact
"independent re-derivation as a certificate" pattern across all 8 categories as a single
named framework. The closest conceptual match (Tool Receipts) validates the *shape* of the
idea without providing new machinery beyond what AMDA has already built for math/logic.

## 6. Recommendation

Ranked by (confidence in accuracy-gate impact) ÷ (implementation + risk cost), given the
2026-07-11 deadline. All algorithm sketches below are meant to slot into
`agent/verifiers.py::verify()`'s existing per-category branches.

### 6.1 Do this: Hypothesis-driven actual-invocation fuzzing for code_generation/code_debugging

**Effort: low (hours, not days). Risk: essentially none — pure-Python, no new heavy
dependency, no change to what "pass" already means, only to how hard it is to earn.**

This is the highest-confidence item in this whole research pass because it isn't really a
new *technique* borrowed from a paper — it's a confirmed, mechanical gap in the current
implementation (§1.7): `_run_python` executes the extracted block as a script, and most
extracted code_generation/code_debugging answers are bare function definitions with no call
site, so the function body's actual logic is currently almost never exercised.

Sketch:
```
1. After extract_python_code() succeeds and the AST contains a FunctionDef, pull the
   function's name and parameter list/type hints (ast.walk for the FunctionDef, inspect
   annotations if present; fall back to trying int/float/str/list[int] as a small guessed
   type ladder when there are no hints, mirroring EvalPlus's Table 1 type-aware mutation
   set: int/float, bool, str, list, dict).
2. Write a small harness file: the candidate code + a Hypothesis @given-decorated wrapper
   that calls the function with generated inputs matching the guessed/annotated types,
   inside a try/except that only fails the test on an *unexpected* exception (let
   documented-by-signature exceptions like ValueError-on-empty-list pass if the function
   itself seems to guard for it — or, simpler for a first cut, just record "did it crash"
   and let that alone gate pass/fail, same binary shape as today's check).
3. Run the harness via the existing _run_python-style subprocess with the existing 5s
   timeout. No network, no extra model call.
4. verify() keeps returning "pass"/"fail" exactly as now; only the rigor behind "pass"
   changes.
```
This directly strengthens the category with the currently-weakest deterministic check
short of "does it parse," using infrastructure the codebase already has 90% of
(`extract_python_code`, `_run_python`, `tempfile`/`subprocess` patterns already present).

### 6.2 Worth trying next: one extra local-model call to generate spec-derived assertions (CRITIC/CodeT-lite)

**Effort: low-medium (needs a new prompt template + a parser for `assert` lines + wiring
one more `local_llm.complete()` call into the code branch). Risk: low-medium — the CRITIC
paper's own ablation (`research/program_aided_verification_patterns.md` §2, already in this
repo's corpus) shows *ungrounded* self-critique can make things worse, but this design is
execution-grounded (the assertions are actually run against actual code), which is
precisely the condition CRITIC's ablation found to help, not hurt.**

Sketch:
```
1. Prompt the local model, given ONLY the task's natural-language spec (not the candidate
   code, to reduce the "same misunderstanding leaks into both" correlated-error risk the
   program_aided_verification_patterns.md research flagged as an open gap in the
   literature — no paper was found either confirming or ruling out this risk, so keeping
   the two generations independent is the cheap mitigation available): "write 3 short
   Python assert statements that check <spec>. Assume a function named f(...). One
   assert per line, no explanation."
2. Parse lines matching `^assert `, keep syntactically valid ones (ast.parse each).
3. Splice: candidate function body + generated asserts, run via existing _run_python
   machinery.
4. All asserts pass -> pass; any assert fails or raises -> fail; no parseable asserts
   extracted -> fall back to current "unknown"/parse-only behavior (never regress below
   today's floor).
```
Cost: one small extra local generation (zero scored tokens — still free under the scoring
rule) plus the same sub-5s execution budget as 6.1. Combine 6.1 and 6.2 in either order;
they're complementary (crash-finding vs. logic-checking).

### 6.3 Consider if time remains: DIY SummaC-lite for text_summarisation, with an ONNX-first implementation to dodge the torch dependency

**Effort: medium (new dependency, needs empirical RAM/latency validation inside the actual
constrained Docker image before trusting it — same "needs an empirical constrained-Docker
test" caveat VERDICTS.md V20 already applies to the primary model). Risk: medium — real
accuracy payoff is the least certain of the three recommended items, and it's the only one
that touches the dependency footprint the intro section flags as a first-order risk.**

If pursued: prefer `onnxruntime` (a lightweight, pure-C++-backed CPU runtime, no torch) over
`transformers`+`torch` if an ONNX export of a small NLI cross-encoder (~22-67M params,
§2.5) can be sourced or exported via `optimum` ahead of time and baked into the image the
same way the GGUF weights already are — this avoids adding several hundred MB of PyTorch
runtime weight for a model that's itself under 100MB. Sketch (SummaC-ZS pattern, verified
algorithm shape from the SummaC paper/repo, §2.3):
```
1. Split the generated summary into sentences (simple regex/NLTK-free splitter — no need
   for a model here).
2. For each summary sentence, score entailment against the source document (or its
   sentences/chunks, whichever fits the NLI model's context window) using the small NLI
   model; take the max entailment probability across source chunks per summary sentence
   (SummaC-ZS's own aggregation rule).
3. Aggregate per-sentence scores (mean or min) into one faithfulness score; threshold to
   pass/fail, replacing the current text_summarisation branch's "unknown" fallback for the
   faithfulness dimension specifically (keep the existing word-limit/one-sentence format
   checks as-is — they're free and orthogonal).
4. MUST be validated empirically (a handful of known-good vs known-hallucinated summary
   pairs, run inside a --memory=4g --cpus=2 container) before trusting the threshold —
   this research pass found no AMDA-specific accuracy number for any of these models, only
   general-benchmark numbers that may not transfer to this project's summarisation prompt
   style (short, word-limited, single-document).
```

### 6.4 Do not implement before the deadline

- **Full Metamorphic Prompt Testing (5 generations), CodeT/AlphaCode-style multi-candidate
  consensus, QAFactEval, SelfCheckGPT/ASC-scale self-consistency for factual QA, FActScore
  proper, Proof-Carrying Certificates/Lean 4** — each is either budget-incompatible (too
  many extra generations for a 30s/task, 2-vCPU box), requires infrastructure AMDA is
  explicitly barred from (external retrieval/knowledge base), or requires a
  disproportionate new-skill investment (formal verification tooling) for the time
  remaining. Listed explicitly rather than silently dropped, per the instruction to be
  honest about what's not worth doing.
- **factual_knowledge verification generally**: no retrieval-free technique was found in
  this pass with a credible path to a calibrated pass/fail signal inside the time and
  latency budget. Given the in-repo precedent that a *free* confidence proxy (logprobs)
  for these open categories already failed to calibrate on a 26-task sample
  (`research/VERDICTS.md` V17), spending remaining pre-deadline time on a *costlier*
  self-consistency proxy for the same category is a weak bet relative to 6.1/6.2, which
  have no calibration question at all (they're deterministic pass/fail on executable
  logic, not a threshold tuned on a small sample).
