IMPORTANT CAVEATS (read before trusting the raw numbers above):

1. code_generation/code_debugging show 0%/4% strict, 100%/96% "unsure" --
   this is NOT a real capability gap. eval/judge.py's _judge_code always
   returns "unsure, semantics unchecked" for any code that merely parses --
   the independent judge has no gold test cases to check actual correctness
   against, by design. This number cannot move without a fundamentally
   different judging methodology (see research/certificate_carrying_answers.md
   for what that might look like -- property tests, spec-derived assertions).
   The agent-side V21 fix improves what ships as "pass" internally, but this
   harness structurally cannot show that improvement as a score delta.

2. logical_reasoning's 57% strict / solver route's 5-pass-6-fail is
   ARTIFICIALLY LOW, for TWO independent, now-fixed reasons (VERDICTS V22,
   all three rounds):
   (a) agent/verifiers.py::format_assignment hardcoded "Position N" instead
       of the puzzle's own label vocabulary (day names/ordinals/"Seat N") --
       every one of the original 8 non-pass "solver" tasks had the exactly
       correct assignment, wrong only in label wording.
   (b) eval/judge.py's own _judge_short_text diluted the overlap score for
       9/37 gemini-sourced gold answers that append a "Justification: ..."
       paragraph the agent was never asked to reproduce -- a judge bug, not
       an agent bug, that persisted even after (a) was fixed.
   With BOTH fixed: re-judging the same (already fixed) round-2 answers
   (research/benchmark_2026-07-09/recheck_logical_reasoning_v2.json) gives
   logical_reasoning 28/37 = 76% strict, solver route 12/13 pass (92%),
   up from 57% / 5-13. Re-judging the ORIGINAL full-228 answers (pre-V22,
   judge-fix only) moves logical_reasoning from 57% to just 59% -- most of
   the gain needed the actual V22 answer-format fix, not just the judge fix
   alone. See VERDICTS.md V22 for the full three-round trace. Remaining 4
   fails in the recheck are grok-sourced gold answers that are themselves
   non-specific placeholders ("Specific arrangement satisfying all
   constraints.") -- a dev-set data quality issue, not agent or judge.

3. This was run on a GPU pod at full speed (-ngl 99), not the 2vCPU/4GB
   grading constraint. Wall-clock/route-mix here should not be read as
   "how fast this would run in the real grading container" -- that's a
   separate, already-answered question (VERDICTS V20, constrained-Docker
   tests). This benchmark's purpose was accuracy/logic validation only.

4. Remote escalations went through Cerebras (gemma-4-31b), a stand-in for
   the real Fireworks-hosted models named in ALLOWED_MODELS. Directionally
   informative for whether the escalation/verification architecture works,
   not a prediction of real-Fireworks-model accuracy.
